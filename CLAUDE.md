## Data pipeline

`public/data/girls.json` and `boys.json` are generated by `scripts/process-data.cjs`, which theoretically downloads the SSA baby names zip and processes it. Actually, someone has to manually grab the zip file from the SSA website and bring it to the right folder, because the SSA blocks all our automated attempts.

These files are committed and must be regenerated manually when new SSA data is released. Each file has the shape:

```js
{
  years: [1925, ..., 2025],
  data: {
    "Ashley": {
      "1992": [rank, count, pct],  // pct = % of all babies that gender that year
      ...
    }
  }
}
```

Only the top 1000 names per gender per year are stored. Names not in the top 1000 in a given year simply have no entry for that year (not a null entry).

## Architecture

**Query model:** The app state is a single `query` object `{ name, refYear, compareYear }` kept in sync with the URL (`?name=Lucy&ref=2025&compare=1992`). `refYear` = the year to look the name up in; `compareYear` = the year to find comparables in. These can be the same year or different.

**Data flow:** `App` loads both JSON files and passes raw data down. All computation happens synchronously in `Results.jsx` on each render â€” there is no caching layer. `Results` calls the utility functions, assembles per-gender section props, and renders `NameSection` for each gender that matched.

**`src/utils/nameData.js`** is the core algorithm file. Key functions:

- `buildByYear(nameData)` â€” inverts the data structure to `{ year: [{name, rank, count, pct}] }`, used for "similarly common" lookup
- `findComparables(targetPct, year, byYear, excludeName)` â€” finds names with the closest pct in a given year
- `findSimilarTrajectory(targetYearData, refYear, allNameData, compareYear, ...)` â€” finds names whose 15-year trailing popularity window most resembles the target's, searching Â±5 years around `compareYear`. Used for the "similarly trending" display tab.
- `findBestAnalogues(targetYearData, refYear, allNameData, ...)` â€” like `findSimilarTrajectory` but sweeps all years 1940â€“2015 to find the best historical matches regardless of era. Used for prediction only (maxAnchor=2015 ensures 10 years of future data exist).
- `computePrediction(targetYearData, refYear, allNameData, matches)` â€” uses analogue names' post-match trajectories to project the target name 5 years forward via score-weighted average of forward ratios.

Both trajectory functions use **single-reference-scale normalization**: candidate windows are divided by the *target's* mean popularity, not their own. This means a name at 0.1% won't match a name at 1.0% even if their shapes are identical.

**`scripts/eval.mjs`** is a dev tool for measuring prediction accuracy. It uses a seeded PRNG (`0xDEADBEEF`) to pick the same 25 names per gender every run, then runs the full pipeline with window sizes [5, 10, 15, 20, 25] and reports MAPE by year as CSV. Run as `npm run eval > tmp/results.csv`. The same seed is used in the browser's (now-removed) `window.runPredictionEval`.

**Deployment:** Netlify, auto-deploys on push to `main`. Build command is `npm run build`, publish dir is `dist`. The `[[redirects]]` rule in `netlify.toml` enables client-side routing.

## UI notes

- All styles are in a single `src/App.css`. No CSS modules or Tailwind.
- Colors: girls = `#ed625d`, boys = `#099fb7`. Prediction line uses lighter variants `#f5b5b0` / `#6cc8d8`.
- `Sparkline` is a hand-rolled SVG component (no Recharts). `NameChart` uses Recharts `LineChart`.
- The chart renders two `<Line>` components on merged data: `dataKey="pct"` (solid, historical) and `dataKey="predicted"` (dashed). They share a single data array where each point has both fields, with the connection point (refYear) having both set to the same value.
- `NameSection` manages the "Similarly Common" / "Similarly Trending" tab state. When `trajectoryMatches` is empty, the trending tab is disabled with a click/hover tooltip explaining why.
